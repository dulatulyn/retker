# 🧠 retker AI — «мозг» системы

AI-слой retker: связывает **обученную модель**, **логи организации**, **RAG-базу знаний**
и **рассуждающего агента с инструментами** за единым фасадом с **фолбеком провайдеров**.

Живёт как самодостаточный подпакет `backend/app/ai/` — «микросервис внутри бэкенда».
Не зависит от внешних SDK (только `httpx`), деградирует в offline-шаблоны, если сети нет.

---

## Зачем он нужен

Детекторы (правила) и модель помечают подозрительное. Дальше нужен слой, который:

- **объясняет** каждую угрозу на русском, опираясь на доказательства и базу знаний;
- **коррелирует** разрозненные алерты в одну историю-инцидент;
- **отвечает на вопросы** аналитика по логам на естественном языке (агент + инструменты);
- **готовит отчёт** по инциденту.

При этом он **не привязан к одному вендору** (Gemini/OpenAI/Anthropic взаимозаменяемы) и
**не ломает демо**, если сеть/провайдер упали — переходит на детерминированные шаблоны.

## Архитектура

```
                          app/ai (фасад: __init__.py)
                                     │
       ┌─────────────┬───────────────┼───────────────┬──────────────┐
       ▼             ▼               ▼               ▼              ▼
   analyst.py     agent.py        tools.py         rag.py        model.py
  (сценарии:   (цикл рассуждения  (поиск по        (TF-IDF по    (sklearn-
   объяснить,    с tool-use)       логам, инциденты, плейбукам +   модель фрода,
   коррелир.,        │             статистика,     память орг)    метрики)
   чат, отчёт)       │             RAG, скоринг)        ▲              ▲
       │             │                 │               │              │
       └──────┬──────┘                 └──────── store (логи/инциденты/орг) ──┘
              ▼
```

**Принцип:** правила и модели — фундамент детекции; LLM — слой повествования поверх,
заземлённый на реальные данные (логи через инструменты, знания через RAG).

## Фолбек-цепочка провайдеров

Порядок по умолчанию (настраивается `AI_PROVIDER_ORDER`):

| # | Провайдер | Модель по умолчанию | Роль |
|---|-----------|---------------------|------|
| 1 | **Gemini** | `gemini-2.5-flash-lite` | основной, дёшево |
| 2 | **OpenAI** | `gpt-4o-mini` | запасной, дёшево |
| 3 | **Anthropic** | `claude-haiku-4-5` | запасной |
| 4 | **offline** | — | детерминированные шаблоны (демо не падает) |

`LLMRouter` идёт по списку: первый доступный (есть ключ) и ответивший — побеждает. Падение
или пустой ответ → следующий. Если ответили никто (или `AI_OFFLINE=1`) → `analyst` отдаёт
шаблонный результат. Так Claude/GPT не тратятся, пока жив Gemini, но подстраховывают.

## Компоненты

| Файл | Ответственность |
|------|-----------------|
| `settings.py` | Конфиг: порядок фолбека, модели, ключи (из `.env`/env), тайминги, offline-флаг |
| `providers.py` | `LLMProvider` (Gemini/OpenAI/Anthropic через REST) + `LLMRouter` с фолбеком |
| `rag.py` | TF-IDF ретрив по плейбукам (`knowledge/*.md`) + память по `org_id` |
| `model.py` | Низкоуровневая загрузка артефактов `ml/*.joblib` + карточки метрик (кэш) |
| `features.py` | Извлечение признаков из логов: транзакции + поведенческий базлайн юзера (контракт колонок) |
| `scoring.py` | Контракт скоринга: `FraudScorer`/`AnomalyScorer` + реестр + объяснимость; модель плагинится через `ml/*.joblib` |
| `tools.py` | Инструменты агента: поиск по логам, инциденты, статистика, RAG, скоринг |
| `prompts.py` | Системные промпты (рус.) и сборка описания инструментов |
| `agent.py` | Цикл рассуждения с tool-use (JSON-протокол, провайдер-агностичный) |
| `analyst.py` | Высокоуровневые сценарии + offline-фолбек |
| `report_export.py` | Рендер отчёта инцидента в `.docx` (python-docx) и `.xlsx` (xlsxwriter) |
| `knowledge/*.md` | База знаний RAG: типы угроз, что делать, типологии отмыва, регуляторика РК |

## Публичный API (зеркалит `brain.py` → drop-in замена)

```python
from app import ai

ai.explain(alert)                 # -> str            объяснение алерта на русском
ai.build_incident(entity, alerts) # -> dict           корреляция в инцидент
ai.nl_query(org_id, q)            # -> {summary, count, events}   NL-запрос по логам
ai.chat(org_id, message, history) # -> {reply, suggestions}      чат-агент с инструментами
ai.report(incident)              # -> {markdown, sections, ...}  отчёт
ai.triage(alert)                 # -> {severity, category, is_true_positive, ...}
ai.score_event(event, org_id)    # -> скоринг события обученной моделью (+ объяснение)
ai.score_features(model, feats)  # -> скоринг по сырым признакам (fraud|anomaly)
ai.record_incident(org_id, inc)  # -> положить инцидент в память орг
ai.record_feedback(org_id, alert, verdict, note)  # вердикт FP/легит → память орг
ai.set_org_context(org_id, text) # -> профиль орг в память
ai.export_report(report, out_dir, base)  # -> {docx, xlsx, markdown}  файлы отчёта
ai.health()                      # -> статус провайдеров / скореров / RAG
```

**Экспорт отчёта (кульминация демо):** `analyst.report(inc)` даёт markdown + секции;
`report_export.export(report, out_dir)` рендерит из них **`.docx`** (резюме, хронология,
гипотеза, рекомендации, MITRE, таблица алертов + строка соответствия закону РК) и **`.xlsx`**
(листы «Инцидент», «Алерты», «Хронология»). Без `python-docx`/`xlsxwriter` — мягко отдаёт `None`.

Интеграция: достаточно в роутере заменить `from .brain import ...` на `from .ai import ...`
(сигнатуры совпадают). Пока другой агент держит `brain.py` — `ai` живёт рядом, не конфликтуя.

## Поток данных

**Чат / NL-запрос (агент с инструментами):**
```
вопрос → agent → LLM решает: вызвать инструмент или ответить
        → tools.search_logs / get_stats / search_knowledge ... (читают store/RAG)
        → наблюдение возвращается в LLM → ... → финальный ответ (заземлён на данные)
```

**Объяснение / корреляция / отчёт (одношаговый structured-вызов):**
```
алерт+доказательства (+ RAG-контекст) → LLM (structured JSON) → разобрали → результат
            └── при ошибке провайдера → детерминированный шаблон
```

## Инструменты агента

| Инструмент | Что делает |
|------------|------------|
| `get_stats` | сводка: события/алерты/инциденты, разбивка по категориям |
| `search_logs` | фильтр событий (класс, страна, юзер, ночь, severity, лимит) |
| `list_incidents` | список инцидентов (статус, лимит) |
| `get_incident` | детали инцидента (таймлайн, гипотеза, рекомендации) |
| `get_alerts` | алерты по категории |
| `search_knowledge` | RAG-поиск по плейбукам (заземление объяснений) |
| `score_transaction` | скоринг по набору признаков (модель `fraud`/`anomaly`) |
| `score_event` | скоринг события по `event_id`: фичи из лога → модель → объяснение |

Инструменты читают только данные **своей организации** (`org_id`) — изоляция тенанта.

## RAG (заземление)

- **Плейбуки** (`knowledge/*.md`) — короткие тексты: что за угроза, чем опасна, что делать,
  типологии отмыва, сниппеты регуляторики РК. Ретрив — TF-IDF на чистом Python.
- **Память орг (по `org_id`)** — структурированная, наполняется по ходу работы:
  - `record_incident(org_id, incident)` — резюме/гипотеза/статус инцидента;
  - `record_feedback(org_id, alert, verdict, note)` — вердикт офицера FP/легит;
  - `set_org_context(org_id, text)` — профиль орг (роли, окна обслуживания).
  Подмешивается в промпт через `rag.context_for(org_id, query)` → меньше повторных ложняков,
  анализ в контексте именно этой организации.
- Логи и числа в RAG **не кладём** — они в `store`, достаются инструментами.

## Модели и контракт скоринга («вставил модель → заработало»)

**Основная модель — единая `threat`** (`ml/threat_model.joblib`, карточка `ml/MODEL.md`):
одна обученная модель на 3 вектора (транзакции / доступ / фишинг) с `event_class`-маршрутизацией.
Метрики hold-out: **overall ROC-AUC 0.982**, транзакции 0.9998, доступ 0.9999, фишинг 0.902.
`UnifiedThreatScorer` стоит первым в реестре → `score_event` использует именно её, с человеческим
объяснением. Если артефакта нет — слой деградирует на правила/шаблоны.

Слой `scoring.py` построен так, чтобы обученную модель достаточно было **положить артефактом
в `ml/` — и она подхватится**, без правок логики:

1. Обучаешь модель на признаках из `features.py` (`FRAUD_FEATURES` / `ANOMALY_FEATURES`) —
   на DataFrame с этими именами колонок (тогда у модели появится `feature_names_in_`).
2. Сохраняешь артефакт:
   - фрод/отмыв → `ml/fraud_model.joblib` (`FraudScorer`, event_class `transaction`);
   - аномалии → `ml/anomaly_model.joblib` (`AnomalyScorer`, `data_activity`/`access`).
3. Готово. `score_event(event, org_id)` сам выберет скорер по классу события, извлечёт
   признаки из лога (`features.py`), выровняет их по `feature_names_in_` и вернёт
   `{score, label, reasons, explanation_ru, ...}`.

Поддерживается `predict_proba` (классификатор) и `decision_function`/`score_samples`
(IsolationForest) — нормализуется в 0..1.

**Совместимость:** если признаки события не пересекаются с колонками обученной модели,
результат помечается `compatible: false` + подсказка переобучить. Так сейчас с текущим
`fraud_model.joblib` (обучен на `creditcardfraud`, признаки `V1..V28` нечитаемы). После
переобучения на читаемых колонках (`FRAUD_FEATURES`, напр. из PaySim) — `compatible: true`,
а `reasons` уже сейчас человеческие («обналичивание средств, обнуление баланса отправителя»).

Метрики (`ml/fraud_metrics.json`) отдаются в `health().model_card` для слайда «под капотом».
Без `joblib`/`scikit-learn` — мягкая деградация (скор `None`), остальной AI работает.

## Конфигурация (env / `backend/.env`)

| Переменная | По умолчанию | Назначение |
|------------|--------------|------------|
| `GEMINI_API_KEY` / `OPENAI_API_KEY` / `ANTHROPIC_API_KEY` | — | ключи провайдеров |
| `AI_PROVIDER_ORDER` | `gemini,openai,anthropic` | порядок фолбека |
| `GEMINI_MODEL` / `OPENAI_MODEL` / `ANTHROPIC_MODEL` | flash / 4o-mini / haiku | модели |
| `AI_OFFLINE` | `0` | `1` → только шаблоны, без сети |
| `AI_HTTP_TIMEOUT` | `30` | таймаут запроса, с |
| `AI_MAX_TOKENS` | `1024` | лимит вывода |
| `AI_TEMPERATURE` | `0.3` | температура |
| `AI_AGENT_MAX_STEPS` | `5` | макс. шагов рассуждения агента |

Ключи берутся из реального окружения, иначе из `backend/.env` (не коммитится — см. `.gitignore`).

## Как расширять

- **Новый провайдер:** добавить класс в `providers.py` + строку в `_REGISTRY`, указать в `AI_PROVIDER_ORDER`.
- **Новый инструмент:** добавить функцию + запись в `TOOLS` (`tools.py`) — агент подхватит из промпта.
- **Новый плейбук:** положить `.md` в `knowledge/` — RAG переиндексирует при старте.

## Зависимости

- Обязательные: `httpx` (уже в `requirements.txt`).
- Опциональные (скоринг моделью): `scikit-learn`, `joblib`, `numpy`.
- Опциональные (экспорт отчёта): `python-docx`, `xlsxwriter`.
- Всё в `requirements-ai.txt`. Без них AI работает, кроме скоринга / экспорта файлов.

## Безопасность

Ключи — только в `backend/.env` (в `.gitignore`), не в коде и не в гите. Если ключ
засветился (например, в переписке) — **ротировать** в консоли провайдера.
