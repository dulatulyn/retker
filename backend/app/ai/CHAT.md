# AI-аналитик — диалоговый чат (агент + стрим + rich-блоки)

Чат-ассистент в дашборде (и живое демо на лендинге). **Изолирован** в двух новых файлах —
не трогает «мозг» и общий `routers.py`, чтобы не мешать параллельной работе:

| Слой | Файлы |
|---|---|
| Бэк: стрим-агент + история | `backend/app/ai/chat_stream.py` |
| Бэк: WebSocket + REST истории | `backend/app/chat_routes.py` (подключён в `main.py`) |
| Фронт: панель чата | `frontend/src/components/dash/AiChatPanel.tsx` |
| Фронт: rich-блоки данных | `frontend/src/components/dash/ChatToolBlock.tsx` |
| Фронт: рендер markdown | `frontend/src/components/dash/Markdown.tsx` |
| Фронт: демо на лендинге | `frontend/src/components/LiveConsole.tsx` |

Инструменты агента — те же, что в `ai/tools.py` (get_stats, search_logs, list_incidents,
get_incident, get_alerts, search_knowledge, score_*).

---

## 1. Транспорт — WebSocket (не SSE)

Поток событий (`/v1/stream`) — однонаправленный, поэтому SSE. **Чат двунаправленный + стрим
токенов → WebSocket.**

`WS /v1/chat/ws?token=<jwt>` — браузерный WebSocket не шлёт `Authorization`, поэтому JWT в query
(как у SSE-стрима). Клиент шлёт `{ message, chat_id }`. Сервер шлёт события:

| event | payload | смысл |
|---|---|---|
| `chat` | `{chat_id, title}` | назначен/выбран диалог |
| `tool` | `{tool, args}` | агент вызвал инструмент (UI: «синяя пилюля») |
| `tool_result` | `{tool, data}` | **данные** инструмента → фронт рисует таблицу/логи/диаграмму |
| `delta` | `{text}` | токены ответа (стрим «по буквам») |
| `done` | `{suggestions, online, provider, elapsed, tools}` | финал |
| `error` | `{message}` | ошибка |

Синхронный генератор `stream_chat` крутится в треде, события проксируются в asyncio через
`_bridge` (`loop.call_soon_threadsafe` + `asyncio.Queue`) — event loop не блокируется.

**Фолбэк:** если WS не поднялся (старый бэкенд без роута, прокси и т.п.) — фронт бьёт
`POST /v1/chat` (нестрим) и помечает ответ плашкой «упрощённый ответ: WebSocket не подключился».

## 2. Агентность «по запросу» (`_preroute` / `_log_args`)

Проблема: LLM на «дай логи» нередко **отказывается** («не могу предоставить логи»), не вызвав
инструмент. Решение — детерминированный pre-routing: по смыслу запроса **принудительно** дёргаем
нужный инструмент И извлекаем из формулировки конкретные фильтры.

- интенты: логи/события/входы/выгрузка/транзакции/почта → `search_logs`; инциденты →
  `list_incidents`; утечки/ИИН/DLP → `get_alerts(leak)`; фишинг/домен → `get_alerts(phishing)`;
  фрод → `get_alerts(fraud)`; график/статистика/сколько → `get_stats`.
- `_log_args` тянет фильтры: **пользователь** `u.berik` (regex `_USER_RE`; домены вроде
  `kaspi-bonus.xn--80a.tk` отсекаются), **страна** (корея/сеул→KR, россия→RU, казахстан→KZ…),
  **действие** (неуспешные→`login_failed`, успешные→`login_success`), **класс события**,
  `night`, `min_severity` (критич/высок→≥4). `list_incidents` тянет `status`.
- если интент НЕ распознан и LLM доступен — обычный агентный цикл (LLM сам выбирает инструменты,
  json-decision через `agent_system`).

> Пример: «дай логи где появляются u.berik» → `search_logs(user="u.berik")` → только его события,
> а не весь хвост логов.

---

## 3. Работает БЕЗ LLM

Инструменты локальные (БД в памяти) → **данные и rich-блоки показываются всегда**, LLM лишь
добавляет прозу-рассуждение. Если LLM недоступен (квота/офлайн/нет ключа) — короткий ответ
собирается детерминированно из данных (`_deterministic_summary`), **по запросу, а не одинаковой
заглушкой**. Признак «бот отвечает одинаково ко всему» = LLM лёг (видно по `analyst._chat_fallback`).

---

## 4. Rich-блоки в чате (`ChatToolBlock`)

`tool_result` рисуется прямо в диалоге, над текстом ответа:

- `get_stats` → плитки (события/алерты/инциденты) + **мини-бар-чарт** по категориям угроз;
- `list_incidents` / `get_alerts` → строки с severity-точкой, score, статусом/категорией;
- `search_logs` → **лог-строки** (mono): время · актор · действие · страна;
- `get_incident` → карточка с таймлайном;
- в шапке блока — ссылка **«перейти»** → `onNavigate(view)` открывает раздел дашборда
  (логи→События, инциденты→Инциденты, сводка→Обзор).

Markdown ответа рендерит свой лёгкий `Markdown.tsx` (без зависимостей): `**жирный**`, `*курсив*`,
`` `код` ``, блоки ```, списки, заголовки, ссылки. Незакрытый markdown во время стрима не ломается.

---

## 5. Стрим в реальном времени — ключевой нюанс Gemini

2.5-модели (`gemini-2.5-flash-lite`) по умолчанию «думают» молча ~10с перед выводом → стрим
выглядел как «нет ответа», потом всё вываливалось разом. **Лечится** `generationConfig.thinkingConfig.thinkingBudget = 0`
(TTFT 11с→~4с, токены идут сразу). Финальный ответ короткий (`max_tokens≈400`, markdown).
Если провайдер не стримит — `chunk_text` режет готовый ответ по словам (псевдо-стрим).

Порядок провайдеров — `AI_PROVIDER_ORDER` (по умолчанию gemini→openai→anthropic), ключи в
`backend/.env`. Провайдеры со стримом реализованы прямо в `chat_stream.py` (`_stream_openai/_anthropic/_gemini`
через `httpx.stream`), чтобы не редактировать `providers.py`.

---

## 6. История чатов

In-memory по `org_id` (в `chat_stream.py`): `create/get/list/delete/append_message`, авто-заголовок
из первого сообщения. REST: `GET/POST /v1/chats`, `GET/DELETE /v1/chats/:id`. В панели — сайдбар
истории (иконка часов), переключение/удаление/новый чат.

---

## 7. Живое демо на лендинге

`LiveConsole.tsx` (терминальный вид) стримит с реального бэка по тому же WS через **публичный**
`GET /v1/chat/demo-token` (токен демо-организации, read-only демо-данные). Если бэк недоступен —
мягкий фолбэк на заготовленные ответы, чтобы лендинг не «ломался». Демо-токен публичный — ок для
хакатона, не для прода.

---

## 8. Что проверять при «не работает»

1. `curl -s localhost:8000/v1/chat/demo-token` → `{token:…}`. Если `404` — бэкенд не перезапущен
   (или старый процесс держит порт: `lsof -ti tcp:8000 | xargs kill -9`).
2. DevTools → Network → WS: при вопросе должно появиться `chat/ws` с кадрами. Нет — фронт на
   REST-фолбэке (увидишь плашку «упрощённый ответ»).
3. Один и тот же ответ на всё = LLM недоступен (квота Gemini сбрасывается раз в сутки; поставь
   рабочий ключ в `backend/.env`). Данные-блоки при этом всё равно показываются.
