# Семантический поиск по логам — техдок

Мини search-engine по событиям организации в стиле SIEM/Elastic. Отдельный движок
`backend/app/ai/search.py`, эндпоинт `GET /v1/search`, UI — выпадающая панель в дашборде
(`components/dash/GlobalSearch.tsx`).

| | |
|---|---|
| **Тип** | гибридный лексический (BM25) + русское глоссирование + интент-синонимы |
| **Скоуп** | события одной организации (`store.events[org_id]`), мультитенант |
| **Зависимости** | нет (чистый Python) → детерминированно, ~<1 мс на демо-объёме |
| **Эндпоинт** | `GET /v1/search?q=&limit=` (Bearer-токен дашборда) |

---

## 1. Зачем не просто `includes()`

Наивный подстрочный поиск (как был в старом `GlobalSearch`) не находит по смыслу: запрос
«обнал ночью» не матчит событие `event_class=transaction, action=cash_out`, потому что слова
разные и язык разный (русский запрос — английские поля). Нужен ранжированный поиск, который:

- понимает **русские запросы по английским полям** (вход→access, обнал→cash_out);
- **ранжирует** релевантность (BM25), а не просто фильтрует;
- даёт **счётчик** и **классификацию** совпадений (как aggregations в Elastic);
- работает **по всем логам орга**, а не только по тому, что подгружено на фронт.

---

## 2. Конвейер

```
запрос q
  │  _expand: токенизация + интент-синонимы + флаг «ночь»
  ▼
для каждого события org:
  _doc(e): текстовое представление (поля + русские глоссы класса/детекторов)
  BM25(q_tokens, doc) → score
  + boost по severity (≥4)
  + фильтр «ночь» (если в запросе)
  ▼
сортировка по (score, severity)
  ▼
{ total, took_ms, results[limit], facets }   ← фасеты строятся по ВСЕМ совпадениям
```

---

## 3. Ранжирование: BM25

Для каждого события считается BM25 по токенам запроса:

```
score(q, d) = Σ_t∈q  IDF(t) · ( f(t,d)·(k1+1) ) / ( f(t,d) + k1·(1 − b + b·|d|/avgdl) )
IDF(t) = ln( 1 + (N − df(t) + 0.5) / (df(t) + 0.5) )
```

- `f(t,d)` — частота токена в документе события, `|d|` — длина документа, `avgdl` — средняя
  длина по корпусу орга, `N` — число событий, `df(t)` — в скольких событиях встречается токен.
- Параметры: `k1 = 1.5`, `b = 0.75` (стандартные). Корпус — события орга (десятки–сотни),
  IDF/avgdl считаются на лету (дёшево).
- **Boost критичности:** `score *= 1 + 0.06·severity` при `severity ≥ 4` — критичные совпадения
  всплывают выше при прочих равных.

---

## 4. Русское глоссирование полей (ключ к «семантике»)

Запросы — на русском, поля события — на английском (`access`, `cash_out`, `dlp_iin`). Чтобы
русский запрос матчился, в текстовое представление события `_doc(e)` **домешиваются русские
глоссы**:

- по классу (`_CLASS_GLOSS`): `access → "вход доступ логин авторизация сессия устройство страна"`,
  `transaction → "транзакция перевод платёж обнал вывод средств финанс сумма"`, и т.д.;
- по детекторам (`_DET_GLOSS`): `impossible_travel → "невозможная поездка перемещение скорость"`,
  `dlp_iin → "иин персональные данные утечка"`, `phishing_domain → "фишинг домен омоглиф punycode"`.

Документ события = `event_class + русская метка класса + action + actor(user/account/ip/country)
+ target(resource/url/host) + детекторы + severityN + глоссы`. Поэтому «утечка иин» находит
событие с детектором `dlp_iin`, хотя в самом событии слова «утечка» нет.

---

## 5. Интент-синонимы и фильтр времени

- `_SYNONYMS` расширяют токены запроса: `обнал → "transaction cash_out"`, `вход → "access login"`,
  `утечк → "leak dlp data_activity"`, `фишинг → "email phishing_domain"`, `подбор → "brute_force"`
  и т.д. Так запрос на «своём» языке дотягивается до токенов событий.
- **Фильтр «ночь»:** если в запросе `ноч/ночн/night`, в выдачу попадают только события с часом
  `00–05` (по `ts`). Пример: «обнал ночью» = cash_out + ночное окно.

---

## 6. Классификация (facets / aggregations)

По **всем** совпадениям (не только по странице `limit`) строятся фасеты:

- `by_class` — сколько по каждому типу (`Доступ/вход`, `Транзакция`, `Данные/выгрузка`,
  `Почта/фишинг`), с человеческой меткой;
- `by_severity` — по полосе критичности (`критичный/высокий/средний/низкий`);
- `by_detector` — топ-6 сработавших детекторов.

Это и есть «классификация результатов»: счётчик + разбивка.

## 8. UI (дашборд)

`GlobalSearch` — инлайн-инпут фиксированного размера в шапке (бар **не двигается** при открытии)
+ выпадающая панель под ним. Вызов по **⌘K**. Панель в стиле LinkedIn:

- шапка: «Найдено N по «запрос» · X мс»;
- **группы по классам** (иконка + счётчик): Инциденты (клиентский матч, переход в инцидент),
  затем Доступ/вход, Транзакция, Данные/выгрузка, Почта/фишинг — с результатами внутри
  (бейдж критичности, действие, актор, время, детекторы, score), до 6 на группу + «ещё N →»;
- состояния: примеры запросов (пусто), «Ищу…», «Поиск недоступен» (сервер не отвечает),
  «Ничего не найдено».

Дебаунс 160 мс, закрытие по клику вне/Esc.

## 9. Ограничения и развитие

- **Лексический + глоссы, не эмбеддинги.** «Семантика» достигается глоссами/синонимами, а не
  векторной близостью — новый, не предусмотренный синоним не подтянется. Для демо это надёжно
  (детерминированно, без сети) и закрывает русские запросы по делу.
- **Без пагинации/подсветки**; выдача ограничена `limit`, фасеты — по всем совпадениям.
- **Развитие:** dense-vector rerank на эмбеддингах (Gemini `text-embedding`) с кэшем по `org_id`
  и блендом к BM25 («как Elastic kNN»), с фолбеком на лексику; подсветка совпадений; пагинация.
